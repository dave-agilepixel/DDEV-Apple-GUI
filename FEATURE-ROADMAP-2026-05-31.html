Feature Gap · Prioritization · Build-Next Roadmap

DDEVUI — What's Missing & What's Worth Building

A candid map of what DDEVUI does today versus DDEV's full feature surface, turned into a prioritized list of what to build next. Every candidate is scored on three axes — effort, value to users, and what it would teach you in Swift/SwiftUI — and given a blunt verdict, including the ones worth not building.

Date: 2026-05-31 App: DDEVUI · Swift 6.2 · SwiftUI · macOS 26 Measured against: DDEV v1.25.2 Method: parallel codebase map + DDEV docs/CLI research Lens: personal learning project (per README)

The one-screen verdict

You've already built the 20% of DDEV that delivers 80% of the daily value — lifecycle, config editing, DB import/export, snapshots, add-ons, logs, diagnostics. What's left is mostly long tail: real coverage gaps, but most of them low-value or a poor fit for a GUI. Chasing command-coverage for its own sake is the trap. The wins that matter are few, and most are small.

Build status — 2026-05-31: quick-win cluster shipped ✓

The full quick-win cluster — A1, A2, A3, A4, A5, A11 and B8 — has been implemented on branch feat/part-a-quick-wins (cards below carry an Implemented pill). The work added a non-persisted rich describe -j detail model (Xdebug state, DB credentials, per-service health/ports — deliberately kept out of the on-disk cache so the DB password and ephemeral ports aren't written to disk), a ddev xdebug service method, and a .command Terminal hand-off for shells. All 200 unit tests pass. Interactive/visual QA of the live panels (which require a running project + Docker) is still recommended.

If you do only one thing: ship the quick-win cluster

Seven small features (each ~hours, not days) that compound into a far more useful daily tool: a live Xdebug on/off toggle, an Open / Launch hub (site, Mailpit, service UIs), a copyable DB-credentials panel, a per-service health + localhost-ports view, a DB-drift warning banner, an "Open shell in Terminal" button, and "Edit .ddev/ in your editor" buttons. None require new infrastructure — the data is already in ddev describe -j and the command service pattern already exists.

If you want a real project to learn from: two big bets

1) A MenuBarExtra for quick start/stop/launch without opening the window — this changes how you live with the app and teaches macOS menu-bar app lifecycle. 2) Streaming-process infrastructure (live ddev logs -f) — your current command runner is request/response; building a streaming variant unlocks live logs, ddev share, and mutagen monitor all at once. The third bet, hosting pull/push, is the only feature here with real ROI for your actual agency workflow — but it's provider-specific and destructive-on-push, so it earns its "big bet" label.

Sparring-partner note: the honest framing

This is a learning side project with no users to please but you. So "value to users" here mostly means value to you, day to day, plus what it teaches you. Read it that way. The biggest risk isn't picking the wrong feature — it's building parity for parity's sake: an embedded terminal, a hooks form editor, Dockerfile GUIs. Those are effort sinks dressed up as progress. The "Deliberately skip" section exists to give you permission to not build them.

Recommended build order

  1. The quick-win cluster (A1–A5, A11, B8). A long weekend of small, high-leverage wins. Mostly new DDEVCommandService methods + reading more of describe -j. Biggest daily-usefulness jump for the least code.
  2. Streaming-process infrastructure → live logs (A6). Build a streaming variant of ProcessCommandRunner once; live logs -f is the first payoff. Reusable foundation for A8 (share) and the mutagen monitor.
  3. MenuBarExtra quick-controls (B1). The feature that most changes how you use the app. Real macOS learning (menu-bar lifecycle, background presence). Leans on the data layer you already have.
  4. Hosting pull/push (A7) — only if your real workflow wants it. Genuinely valuable for agency work, but provider-specific and destructive on push. Do it deliberately, with hard confirmation guards, or not at all.
  5. Cherry-pick the NICE-TO-HAVE long tail as the mood takes you. Global config screen, multi-select, command history, favorites, registry browse. Each is self-contained — perfect "I feel like coding tonight" fodder.
  6. Don't build Part C. Seriously. Hand off to Terminal.app / your editor instead.

Part A

DDEV feature-parity gaps

Things DDEV itself does that DDEVUI doesn't yet expose. Ordered roughly by payoff. The app already shells out to ddev and parses the .raw JSON envelope, so most of these are "one more DDEVCommandService method + a control."

QUICK WIN Implemented A1 · Open / Launch hub (site, Mailpit, service UIs)
maps to ddev launch ddev launch -m describe -j → urls / mailpit_url / services.*.http_url
Effort S Value ●●● Learn ●●○

What it is

One obvious place to open every URL a project exposes: the primary site, Mailpit, XHGui, and any add-on service UI (phpMyAdmin, Adminer, etc.). Today you open the primary URL from a menu and XHGui has its own launch — but Mailpit and add-on service URLs aren't surfaced, and the action isn't prominent.

What you'd build

A split "Open" button / menu in ProjectInspectorView driven entirely by the JSON you already fetch: primary_url, mailpit_url, and services.*.http_url from ddev describe -j. No hardcoded ports — read the fields. Opens via the existing MacWorkspaceOpener.

Native-GUI fit

Perfect. Launching URLs is exactly what a desktop app should make one-click. The research flagged "don't hardcode service URLs" — derive them from describe -j so it's correct across project types and add-ons.

Verdict: Build it. Tiny effort, you'll click it ten times a day. Fold the existing "Open Primary URL" menu item into it.
QUICK WIN Implemented A2 · Live Xdebug on/off toggle
maps to ddev xdebug on|off|status
Effort S Value ●●● Learn ●○○

What it is

A real-time toggle. Today the app only exposes Xdebug as a config flag that needs a restart. But ddev xdebug on/off flips it on the running container instantly — and developers flip it constantly, because Xdebug-on makes every request slow.

What you'd build

A SwiftUI Toggle in the inspector bound to xdebug_enabled (already in describe -j), calling new xdebug(on:) methods in DDEVCommandService. No restart, no config edit.

Native-GUI fit

Ideal — a clean on/off/status command is the textbook case for a toggle switch. Same pattern works for XHProf/Blackfire later (see A17).

Verdict: Build it first. Possibly the single highest value-to-effort ratio in this whole document.
QUICK WIN Implemented A3 · Copyable database-credentials panel
maps to describe -j → dbinfo{}
Effort S Value ●●● Learn ●○○

What it is

A little panel showing host, port, db name, user, password and the published host port — each with a one-click copy button. Developers grab these constantly to wire up an external DB client or a config file.

What you'd build

ddev describe -j's dbinfo block already has it all: host, dbname, username, password, dbPort, published_port (the 127.0.0.1:PORT you connect to from the host). Decode it into the DDEVProject/describe model and render a copy-row list in ProjectInspectorView.

Native-GUI fit

Excellent — this is painful on the CLI (you re-run describe and squint) and trivial in a GUI. Pairs naturally with A1 and the "Open in TablePlus/Sequel Ace" launchers you already have.

Verdict: Build it. Cheap, and removes a genuine daily papercut.
QUICK WIN Implemented A4 · Per-service health + localhost ports
maps to describe -j → services{} / router_status / ssh_agent_status
Effort M Value ●●● Learn ●●○

What it is

A services table in the inspector: each container (web, db, xhgui, add-ons) with its image, running/stopped/created status, and the ephemeral 127.0.0.1:PORT Docker assigned — plus router and ssh-agent health. Turns the inspector from "config summary" into "what's actually running right now."

What you'd build

Expand the describe model to decode the services map (short_name, image, status, host_ports_mapping[], host_http_url) plus router_status/ssh_agent_status. Render a compact table. This also surfaces the partial/unhealthy state (e.g. web up, db unhealthy → status_desc: "db:") the current status badge probably flattens.

Native-GUI fit

Strong. A table of live service health is exactly the kind of at-a-glance overview a GUI does better than scrolling CLI output.

Verdict: Build it. Slightly more decoding work than A1–A3, but it's the feature that makes the detail pane feel substantial.
QUICK WIN Implemented A5 · Database-drift warning banner
maps to ddev utility check-db-match get-volume-db-version
Effort S Value ●●○ Learn ●○○

What it is

A proactive banner when the on-disk database volume disagrees with the configured DB type/version — a common, confusing failure mode. You already run check-db-match inside the diagnostics screen; the win is surfacing it as a warning the user doesn't have to go looking for.

What you'd build

Run check-db-match (and optionally get-volume-db-version) on project select; if mismatched, show a yellow banner in the inspector with a one-line explanation and a link to A12 (migrate-database) or the manual fix.

Native-GUI fit

Great — turning a buried diagnostic into an ambient warning is precisely the value a GUI adds over the CLI.

Verdict: Build it. The command already runs in your code; you're just promoting its result to a banner.
QUICK WIN Implemented A11 · "Open shell in Terminal" (ddev ssh hand-off)
maps to ddev ssh ddev mysql
Effort S Value ●●● Learn ●○○

What it is

A button that drops you into a shell inside the web (or db) container — but instead of embedding a terminal, it opens Terminal.app (or iTerm) running ddev ssh in the project directory. 90% of the value for 5% of the effort.

What you'd build

An NSWorkspace/open -a Terminal hand-off that launches the user's terminal with the right working directory and command. Same pattern for ddev mysql/ddev psql REPLs.

Native-GUI fit

Honest fit: interactive shells are a poor fit for an in-app form, and the research called this out as one of the architecturally hard CLI-only commands. The hand-off side-steps the entire PTY problem. (The embedded-terminal version lives in Part C for a reason.)

Verdict: Build the hand-off. Resist the embedded terminal — see C1.
BIG BET A6 · Live / streaming logs (logs -f)
maps to ddev logs -f
Effort M–L Value ●●● Learn ●●●

What it is

Logs that stream live and autoscroll, instead of the current tail-N snapshot. Watching logs while you reproduce a bug is a daily activity.

What you'd build

The hard part isn't the UI — it's that your ProcessCommandRunner is request/response (run, capture, return). logs -f needs a streaming command runner: spawn the process, read stdout incrementally, push lines into the view, support stop/cancel and backpressure. That's a genuinely meaty Swift concurrency exercise (async sequences, task cancellation, off-main reads — the 2026-05-30 audit's H1/H2 territory).

Native-GUI fit

Good — and the streaming subsystem you build here is the shared foundation for ddev share (A8) and ddev mutagen monitor. That's why it's a bet, not just a feature.

Verdict: Build it second, after the quick wins. High learning value precisely because it forces you to solve streaming-process lifecycle properly. Do it once, reuse it three times.
BIG BET A7 · Hosting pull / push (provider sync)
maps to ddev pull <provider> ddev push <provider>
Effort L Value ●●● (for agency work) Learn ●●●

What it is

Pull the database + files down from a hosting provider (Platform.sh, Upsun, Acquia, Pantheon, Lagoon, rsync…), or push up to it. This is the feature that connects local DDEV to a real deployment workflow — the highest real-world ROI item here for your Agile Pixel work.

What you'd build

Detect configured providers by reading .ddev/providers/*.yaml; show available pull/push targets; run ddev pull/push <provider> with --skip-db/--skip-files options and live progress (leans on A6's streaming runner). Push is destructive on the remote — it needs a hard, type-the-project-name confirmation, the way your DB-import drop already does.

Native-GUI fit

Good, with care. The danger is making a destructive remote operation too easy. Treat push like a loaded gun: confirmation, clear "this overwrites PRODUCTION" language, maybe default-hidden.

Verdict: Build it if your actual workflow uses provider sync. If you mostly work locally, this is a lot of provider-specific plumbing for little personal payoff — be honest about which camp you're in.
NICE TO HAVE A8 · ddev share (public tunnel)
maps to ddev share
Effort M Value ●●○ Learn ●●○

What it is

Expose the local project on a public URL (ngrok / cloudflared) for a client demo, with a copy-the-URL button and a stop button.

What you'd build

A long-running process (the tunnel stays open) whose stdout you parse for the public URL — so it depends on A6's streaming runner. Surface URL + provider + a prominent "Stop sharing" control.

Native-GUI fit

OK. Needs persistent-process management and a clear running/stopped state, but the payoff (grab URL, demo, stop) is genuinely nicer in a GUI than babysitting a terminal.

NICE TO HAVE Implemented A9 · Arbitrary command runner (ddev exec)
maps to ddev exec
Effort M Value ●●○ Learn ●●○

What it is

A field to run an arbitrary one-shot command inside the web (or db) container and see the output — the escape hatch for everything you didn't build a button for.

What you'd build

A small input + service picker that pipes into a new exec method and renders results in the existing CommandOutputView. Keep the input-validation hardening you already have (the service rejects dash-prefixed args to prevent flag injection — be careful here since exec is the one place you want to pass through a full command line).

Native-GUI fit

Decent for one-shot exec. Interactive commands belong in the A11 terminal hand-off, not here.

Verdict: Backlog. Useful power-user escape hatch, but most real uses are covered by the framework presets you already ship + A10.
Built 2026-06-02: A "Run Command" console in the Manage tab — service picker (Web/DB) + input, output in the Logs tab. Runs via ddev exec -s <svc> bash -c "<cmd>": the command is a single opaque arg, so the user's own flags/pipes are never parsed as ddev flags (the flag-injection concern is side-stepped, not fought), and shell features work. Process spawns ddev without a shell, so no host-side shell injection either.

Part B

UX & quality wins (not DDEV features)

These aren't DDEV commands — they're ways the app itself could be nicer, several surfaced by the 2026-05-30 code audit. For a learning project, these are where the Swift/SwiftUI learning is richest.

BIG BET B1 · Menu-bar extra (quick controls)
SwiftUI MenuBarExtra
Effort M–L Value ●●● Learn ●●●

What it is

A menu-bar dropdown listing your projects with start/stop/launch right there — without opening the main window. For a local-dev tool you reach for ten times a day, this is arguably the killer feature: it changes the app from "a window you open" to "always-there controls."

What you'd build

A SwiftUI MenuBarExtra scene in DDEVUIApp.swift backed by your existing ProjectDashboardViewModel data. The interesting parts: window-vs-menubar lifecycle, keeping the list fresh (ties into B2), and "launch-at-login / run-in-background" behavior.

Part C

Deliberately skip

Permission to not build. These are real DDEV capabilities, but they're either a poor fit for a GUI, far more effort than payoff, or better handed off to another tool. Listing them as explicit "no" decisions is as valuable as the build list.

SKIP C1 · Embedded in-app terminal (PTY)
do A11 instead ddev ssh ddev mysql
Effort XL Value ●●○

Why skip

An embedded PTY/terminal (SwiftTerm-style) to run ddev ssh/mysql inside the app is a whole subsystem: pseudo-terminal handling, ANSI rendering, resize, signals. The research explicitly flagged interactive shells as the architecturally hard, CLI-native case. A11's Terminal.app hand-off delivers ~90% of the value for ~5% of the effort. Building your own terminal is a project unto itself, not a feature of this app.

Verdict: Don't. Hand off to Terminal.app (A11). Revisit only if "learn to build a terminal emulator" becomes the goal itself.
SKIP C2 · Replicating ddev tui
redundant ddev tui
Value ●○○

Two technical notes the research surfaced

1 · You're already parsing the right thing — keep doing it

DDEV's -j output is a log envelope — the machine-readable payload lives in the .raw key, while .msg is the rendered Unicode table. Your service already decodes this correctly. The point for the roadmap: nearly every quick win (A1, A3, A4, A18) is just decoding more fields from describe -j/version -j you already fetch — not new commands. That's why the cluster is cheap.

2 · The hard architectural seam is streaming, and your audit already pointed at it

The features that don't fit your request/response command runner — live logs (A6), share (A8), mutagen monitor — all need a cancellable, streaming subprocess with proper teardown. The 2026-05-30 audit's H1 (poll loop with no off-switch) and H2 (subprocesses that can't time out or cancel) are the same seam. Fix that infrastructure once and you both close audit findings and unlock the Part-A/B streaming features. Sequence it deliberately rather than bolting streaming onto each feature ad hoc.
