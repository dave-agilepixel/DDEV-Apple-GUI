DDEVUI — Rock-Solid Review

A read-only audit focused on long-running / looping tasks and their teardown, retain cycles & garbage collection, unguarded code, code smells, and performance hot paths. No code was changed. Every finding includes a concrete suggested fix.

Executive summary

The codebase is in genuinely good shape. The single CRITICAL from the previous review (2026-05-28 — the subprocess pipe-deadlock + main-thread block) has been properly fixed: ProcessCommandRunner now drains stdout/stderr concurrently before waitUntilExit() and runs off the main thread. There is no outstanding CRITICAL, no retain cycle, no force-unwrap (!/try!/as!) in the audited code, and the one long-lived task uses [weak self] with a deinit cancel.

The outstanding work clusters around two themes the brief asked about directly: (1) a background poll loop that runs for the entire app session with no off-switch, and (2) subprocesses that cannot be cancelled or time out. Together these are the closest thing to a "task that runs forever / hangs and holds a resource" in the app. Everything else is medium/low: thread-pool pressure, a slowly-growing state dictionary, main-thread file I/O, and a healthy backlog of view-layer performance and code-smell cleanups.

HIGH H1 · Prerequisite poll loop never stops — it spawns subprocesses every 5 s for the entire app session Long-running task
CWE-noise · PrerequisiteMonitor.swift:21–31  ·  ContentView.swift:88–90
RESOLVED 2026-05-31 (reframed) — The headline claim was inaccurate: stop() does exist (PrerequisiteMonitor.swift:33), matching this audit's own "strengths" note. The real defect — the loop never self-terminated and nothing called stop() outside deinit — is fixed: the loop now breaks and clears pollTask once state.allSatisfied (so it stops spawning docker info/ddev version after the gate clears and start() can re-arm), and ContentView now pauses/resumes it via onChange(of: scenePhase). Covered by three new PrerequisiteMonitor lifecycle tests (stops-when-satisfied, keeps-polling-then-stop, re-arm).

What's wrong

PrerequisiteMonitor.start() launches an infinite while !Task.isCancelled loop that calls service.check() (which shells out to docker info + ddev version) and sleeps 5 s. It is started from ContentView's .task { prerequisites.start() } but there is no matching stop() anywhere — no .onDisappear, no scenePhase observation, no "satisfied → stop" condition. The loop only ends when the @StateObject is deallocated (deinit cancels it), which for this single-window app is effectively app quit.

public func start() {
    guard pollTask == nil else { return }
    pollTask = Task { [weak self] in
        guard let self else { return }
        while !Task.isCancelled {            // ← never exits on success
            await self.refresh()              // docker info + ddev version
            guard !Task.isCancelled else { return }
            try? await Task.sleep(for: self.pollInterval)
        }
    }
}

Why it matters

This is exactly the "looping task that isn't cleaned up" the brief targets. It is not a memory leak (the [weak self] capture and deinit cancel are correct), but it is an unbounded-duration background loop doing real work forever: ~12 docker info + 12 ddev version process spawns per minute, indefinitely, even after Docker/DDEV are confirmed healthy and the blocking sheet is dismissed. On a laptop that's continuous wake-ups and battery drain for zero benefit once prerequisites pass. It also amplifies H2: if the Docker daemon hangs, every 5 s the loop launches another docker info that never returns (see H2).

Suggested fix

Back off or stop once satisfied, and tie the lifecycle to scene phase. Minimal change — make the loop self-terminate when prerequisites are met and offer a cheap re-arm:

// Option A — stop polling once everything is satisfied; re-arm on demand.
while !Task.isCancelled {
    await self.refresh()
    if self.state.allSatisfied { break }   // idle until re-armed
    guard !Task.isCancelled else { return }
    try? await Task.sleep(for: self.pollInterval)
}
self.pollTask = nil  // allow start() to relaunch when the user reopens the sheet
// Option B — also pause in the background, in ContentView:
@Environment(\.scenePhase) private var scenePhase
// ...
.onChange(of: scenePhase) { _, phase in
    phase == .active ? prerequisites.start() : prerequisites.stop()
}

Both leave the existing start() idempotency guard and deinit cancel intact.

HIGH H2 · Subprocesses ignore task cancellation and have no timeout — a hung child blocks forever and pins a thread Cancellation / hang
CommandRunning.swift:26–95
RESOLVED 2026-05-31 — ProcessCommandRunner.run now wraps execution in withTaskCancellationHandler and shares the child via a lock-guarded ProcessController: task cancellation throws CancellationError after SIGTERM→SIGKILL, and a new CommandSpec.timeout drives a watchdog that terminates a wedged child (surfaced as a new CommandRunnerError.timedOut). The prerequisite poll's docker info + ddev version now run with a 15 s cap, closing the H1×H2 thread-exhaustion path. Covered by ProcessCommandRunnerTests (cancel terminates in ~0.3 s, timeout in ~1 s).

What's wrong

ProcessCommandRunner.executeBlocking calls process.run() then process.waitUntilExit() with no withTaskCancellationHandler and no watchdog. When the owning Swift Task is cancelled (view dismissed, project switched, app quitting) the child ddev/docker process keeps running to completion. If the child hangs (a ddev start stuck on Docker, a docker info against an unresponsive daemon), the continuation never resumes and one DispatchQueue.global thread is pinned permanently.

Why it matters

Long operations (ddev start/restart routinely take 10–60 s) cannot be aborted at all — there is no cancellation path end-to-end. Worse, this combines with H1: a stalled Docker daemon means the 5 s poll loop launches a fresh docker info every cycle, each one hanging and consuming another global-queue thread. Over a few minutes that exhausts the (small, ~64-thread) GCD pool and the whole app's async machinery stalls. This is the most serious latent failure mode in the app.

Suggested fix

Adopt cancellation + a timeout watchdog in the runner. process.terminate() (SIGTERM), escalate to SIGKILL after a grace period:

return try await withTaskCancellationHandler {
    try await withCheckedThrowingContinuation { cont in
        DispatchQueue.global(qos: .userInitiated).async {
            // ... existing pipe-drain + run ...
            // watchdog:
            let deadline = DispatchTime.now() + .seconds(maxSeconds)
            if readQueue.sync(execute: ...) { /* timed out */ process.terminate() }
        }
    }
} onCancel: {
    process.terminate()                 // SIGTERM on Task cancel
    // optionally schedule SIGKILL after grace
}

Add a per-command timeout cap (e.g. 120 s for mutations, 15 s for the poll's docker info) so a wedged daemon can never pin a thread indefinitely. This is the listed "L4" follow-up from the prior review, promoted to HIGH because of the H1 interaction.

MEDIUM M1 · Every command pins a GCD thread for its full lifetime; describe/read fan-out is uncapped Thread-pool pressure
CommandRunning.swift:28  ·  ProjectDashboardViewModel.swift · enrichProjectsWithDetails
RESOLVED 2026-05-31 — The describe fan-out (the only N-way read burst) is now bounded by a new order-preserving concurrentMap(_:limit:) helper at maxConcurrentDescribes = 4; mutations remain capped at 3 via CommandScheduler. The remaining reads (logs/config/snapshot-list/addon-list) are user-driven single-shots tied to selecting one project, not bursts, so they don't create N concurrent threads. Covered by ConcurrentMapTests (peak ≤ limit, order preserved).

What's wrong

Each run dispatches a fully-blocking body (run + waitUntilExit + group.wait) onto DispatchQueue.global and holds that thread for the whole subprocess. enrichProjectsWithDetails fans out one describe per project via TaskGroup with no concurrency cap, and reads (logs/config/snapshots/add-ons) bypass CommandScheduler entirely. Only mutations are capped (max 3).

Why it matters

A workspace with many projects can put N blocked threads in flight at once (N describes + uncapped reads + the poll loop), pressuring the global pool and risking priority inversion / starvation for other async work. The cap that exists (mutations) doesn't protect the most common burst (a refresh that describes every project).

Suggested fix

Run blocking process work on a dedicated bounded queue owned by ProcessCommandRunner (or an OperationQueue with maxConcurrentOperationCount), and cap the describe fan-out — e.g. chunk the TaskGroup to ~4 in flight, or route reads through the same scheduler with a separate permit pool.

MEDIUM M2 · commandStates dictionary is not pruned on the refresh path — slow unbounded growth Unbounded growth / GC
ProjectDashboardViewModel.swift:931–942 (applyProjects)
RESOLVED 2026-05-31 — applyProjects(_:) now prunes to live keys: commandStates = commandStates.filter { liveIDs.contains($0.key) || $0.value.isBusy }, keeping busy entries so an in-flight command that briefly drops from the list isn't lost. Covered by two tests (stale entry pruned, busy entry retained).

What's wrong

Per-project history is correctly bounded (50 entries × 32 KiB). The explicit trash action prunes its entry (commandStates[removedID] = nil). But applyProjects(_:) — the path used whenever a project disappears via a list refresh (deleted/unlinked in-app, or removed outside the app) — only reassigns self.projects and recomputes selection. It never prunes commandStates.

private func applyProjects(_ projects: [DDEVProject]) {
    self.projects = projects
    // ↑ commandStates entries for vanished projects are never removed here
    if let selectedProjectID, ... { ... } else { ... }
}

Why it matters

Bounded per key, unbounded in key count. Over a long session with project churn, stale entries accumulate (each up to ~50×32 KiB of retained command history). It's a slow leak, not a crash, but it's exactly the "not correctly garbage collected" pattern flagged in the brief.

Suggested fix

Prune to live keys at the end of applyProjects (keep busy entries so an in-flight command that briefly drops from the list isn't lost):

let liveIDs = Set(projects.map(\.id))
commandStates = commandStates.filter { liveIDs.contains($0.key) || $0.value.isBusy }
MEDIUM M3 · Project cache JSON is read/written synchronously on the MainActor Main-thread I/O
ProjectCacheStore.swift  ·  called from ProjectDashboardViewModel (loadCachedProjects / refreshProjectsFromDDEV)
RESOLVED 2026-05-31 — ProjectCacheStoring.loadProjects/saveProjects are now async. The stores are non-actor-isolated, so when the @MainActor view model awaits them the JSON encode/decode + disk I/O run off the main actor instead of stuttering the hot refresh path. All four VM call sites and the tests were migrated to await.

What's wrong

FileProjectCacheStore.loadProjects() / saveProjects() do synchronous Data(contentsOf:) / data.write(to:) plus JSON encode/decode, and are called from the @MainActor view model on every refresh and re-describe.

Why it matters

For a large project list this blocks the main thread on disk I/O during the hot refresh path — a stutter source. Disk I/O on the UI actor is the kind of thing that's invisible on an SSD with 3 projects and very visible with 40 on a busy machine.

Suggested fix

MEDIUM M4 · refresh() has no in-flight guard — overlapping full refreshes can stack Re-entrancy
ProjectDashboardViewModel.swift · refresh / loadCachedProjectsThenRefresh
RESOLVED 2026-05-31 — Guarded at the shared chokepoint: refreshProjectsFromDDEV now early-returns when a private isRefreshInFlight flag is set, serialising every refresh path (user refresh, cache-warm background refresh, mutation-triggered .fullList) so two listProjects + describe fan-outs can't stack. Covered by testOverlappingRefreshIsDroppedByInFlightGuard.

What's wrong

refresh() sets isRunningGlobalCommand = true but does not early-return when already running. The toolbar button is disabled while running (UI mitigation), but loadCachedProjectsThenRefresh() (from .task on appear) and a refresh triggered by a .fullList mutation can run two listProjects + N-project describe fan-outs at once.

MEDIUM M5 · PipeBuffer accumulates unbounded output in memory before truncation Memory spike
CommandRunning.swift:99–110
RESOLVED 2026-05-31 — Pipes are now drained in chunks via availableData and PipeBuffer.appendCapped stops accumulating past a configurable cap (default 4 MiB), while still fully draining the pipe so the child never blocks on a full kernel buffer. Truncation is surfaced with a …[output truncated at N MiB] marker. The cap is injectable via ProcessCommandRunner(maxCapturedBytes:); covered by testLargeOutputIsCappedAndMarkedTruncated.

What's wrong

stdout/stderr are read fully into a Data via readDataToEndOfFile() with no size cap, then converted to String. The deadlock is fixed, but the in-memory buffer is unbounded. History storage is capped at 32 KiB — but only after the entire buffer already exists in RAM.

Why it matters

A runaway ddev logs with a huge tail, a verbose utility diagnose, or import-db progress can briefly hold hundreds of MB (Data + its String copy) and spike RSS.

Suggested fix

Read in chunks (readData(ofLength:) loop or readabilityHandler) and stop appending past a cap (e.g. a few MB), marking the result truncated — mirroring the existing history bound but applied at the source.

MEDIUM M6 · No Docker executable resolver; daemon-down and CLI-missing are indistinguishable Robustness
PrerequisiteService.swift:59–68
RESOLVED 2026-05-31 — Added DockerExecutableResolver mirroring the DDEV one, probing /opt/homebrew/bin, /usr/local/bin, /Applications/Docker.app/Contents/Resources/bin, ~/.docker/bin and OrbStack's ~/.orbstack/bin — known-good locations first (so it's S3-correct from the start) then PATH — and wired its absolute path into dockerDaemonReady(). This fixes the false "Docker missing" under a minimal LaunchServices PATH / non-standard install. The proposed .launchFailed vs .nonZeroExit split was judged unnecessary: every command runs via /usr/bin/env (so a missing binary surfaces as a non-zero exit, not a launch throw), and the UI's CLI-missing-vs-daemon-down distinction already comes from the installed/running runtime-app lookup (.missing vs .notRunning/.starting). What's wrong

dockerDaemonReady() runs docker info by bare name (executable: "docker"), relying entirely on the hardcoded PATH in environmentForGUIApp(). Unlike ddev (which has DDEVExecutableResolver probing known locations), there is no docker resolver. And any throw — launch failure (binary not found) or non-zero exit (daemon down) — is collapsed into the same "not ready".

Why it matters

A GUI app launched by LaunchServices inherits a minimal PATH. Docker installed under a non-standard prefix (custom Homebrew, OrbStack, ~/.docker/bin) yields a false "Docker missing", and the user gets a misleading prerequisite screen with no way to tell "install Docker" from "start Docker".

Suggested fix

Add a DockerExecutableResolver mirroring the DDEV one (probe /opt/homebrew/bin, /usr/local/bin, /Applications/Docker.app/Contents/Resources/bin, OrbStack paths) and pass the absolute path. MEDIUM M9 · God-object view model — ~972 lines, ~22 @Published properties, ~40 methods Architecture / perf
ProjectDashboardViewModel.swift (whole file)

What's wrong

A single ObservableObject owns project list, selection, search, per-project command state, snapshots, logs, config, add-ons, diagnostics, preferences, and installed apps. With ObservableObject/@Published, any change invalidates every view observing the object.

Why it matters

Two costs: (1) broad, unnecessary re-renders (a snapshot list update re-evaluates the diagnostics view's body); (2) it's hard to reason about concurrency when ~40 async methods all touch shared @Published state on one actor — it's the substrate that makes M4/M7/M2 easy to introduce.

Suggested fix

LOW L4 · CommandScheduler.acquire() is non-cancellable; manual acquire/release is leak-prone Concurrency
CommandScheduler.swift:27–35 · manual sites in ProjectDashboardViewModel
RESOLVED 2026-05-31 — acquire() is now async throws and cancellation-aware: a cancelled queued caller is removed from the FIFO queue (tracked by waiter id) and throws CancellationError rather than parking forever (actor isolation + remove-before-resume prevents a double-resume race). What's already done well

Worth stating plainly, because it shapes the priorities — these are not areas that need work:
